Mimic.

Mimic isn't a new framework, it's simply an idea.

Most CSS frameworks are absolutely beautiful and simple to use. However, it takes a lot of work to make it my own. It's hard enough to be creative, but to override and re-work other craftsman's design is quite the challenge. This is probably why we see so many "like-minded" websites.

Like a lot of devs, I have to work with legacy codes. Codes that are written with the best of intentions (initially.) Most of my time is spend hunting down that one rogue class or inherited styles in an endless sea of stylesheets.

I've been there before, starting the project with carefully thought-out SASS structure with sensible, modular approach to writing beautifully-orchestrated symphony of classes and inheritance. It's CSS nirvana. After submitting for review and QA, you start to find holes. Ugly, one-of-a-kind holes that needs to be patched with an equally ugly, non-sensical, duplicating work-arounds and hacks that will ultimately end with the humiliating !important declaration(s).

After going through the same insanity over and over again, I started to think differently. Why not use the DOM class name to build a style? Sure it's little more work, but I'll always be able to control the style by just adding or removing a class name, and not have to touch the stylesheet.

Example:

This would be a usual paragraph with a defined class that specifies a margin-bottom of 20px.

This would be the same paragraph but with a class of "margin-0".

One of the advantages of using a system like Mimic is that you can always tell how the element is going to look without having to render it. If you see this code:

<p class="margin-top-20 bg-color-black color-white align-right">Can you guess my style?</p>

Would you have any question as to how that paragraph would render?

OMG, that's a lot of classes.

Listen, I'm not saying abandon all defined styles. I'm simply saying, make things more modular.

This is especially true when you have to deal with multiple authors in a CMS environment. Most authors don't understanding DOM targeting, nor do they care. Most of the time, they just want to make that one thing go away or be able to add a little more padding (which is why you see so many <br>s.

What about when you can't make changes to the live servers without doing a build? Build usually means QA, and QA usually means more time than what it's worth.

IDK, I'm still not convined.

Which is okay. As I mentioned above, Mimic isn't a framework, it's an idea. It's a chance for you to try something new. It's something that I tried on my own projects and felt that it was worth sharing. Heck, you can even build your own Mimic if you desire.
